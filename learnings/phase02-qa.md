## Python — 객체 시스템 (`__eq__`, `__hash__`, `__repr__`)

### Q: `__hash__`의 구현이 이상해 보여요. `self` 클래스 객체의 메모리 주소 해시가 아니라 본인의 각 변수들의 조합을 해시로 하는 게 뭔가 이상합니다. 그럼 한 클래스에 attribute도 많고 함수도 많으면 `hash(함수들 + attribute)` 형태로 전부 다 반환되는 건가요? 그 이유는요?

#### 해시의 목적부터

파이썬의 `hash()`는 **"이 값을 요약하는 짧은 지문"**입니다. 용도는 단 하나: **dict/set의 버킷을 찾는 것**. dict는 내부적으로 해시 테이블이라, `d[key]` 조회 시 `hash(key)`를 계산해서 "이 키는 몇 번 버킷으로 가라"를 판단합니다. `set`의 `in` 연산자도 동일.

그래서 해시의 유일한 기술적 의무는 딱 하나:

> **"같다고 판정되는 두 객체는 반드시 같은 버킷으로 들어가야 한다"**

이 한 줄이 파이썬 해시 규약의 전부입니다. 공식 표현:

```
a == b  →  hash(a) == hash(b)      (필수)
hash(a) == hash(b)  →  a == b       (필수 아님, 충돌 허용)
```

왼쪽→오른쪽만 강제됩니다. 역은 강제 안 됩니다 — "서로 다른 두 값이 우연히 같은 해시를 갖는 것"은 완전히 허용됩니다(이걸 **해시 충돌**이라고 하고, dict는 충돌 시 버킷 안에서 선형 탐색으로 해결합니다).

#### 그래서 왜 필드 "몇 개만" 쓰는가

**`__eq__`가 비교하는 것과 정확히 같은 필드를 써야 하기 때문**입니다. 이게 위 규약의 직접적 귀결이에요.

voxprep `ListEntry`의 `__eq__`는 이렇게 생겼습니다:

```python
def __eq__(self, other):
    return (
        self.audio_path == other.audio_path
        and self.speaker == other.speaker
        and self.language == other.language
        and self.text == other.text
    )
```

이 정의는 "이 네 필드가 같으면 같다"고 선언합니다. 그럼 해시도 반드시 **이 네 필드만** 가지고 계산해야 합니다. 왜?

- 만약 해시에 **다섯 번째 요소**(예: 생성 시각, 메모리 주소)가 들어가면: 같은 4필드 값을 가진 두 인스턴스가 다른 해시를 가질 수 있음 → `a == b`인데 `hash(a) != hash(b)` → 규약 위반 → dict 조회가 깨짐
- 만약 해시에 **네 필드 중 일부만** 쓰면(예: `audio_path`만): `audio_path`는 같은데 `text`가 다른 두 객체는 해시는 같지만 `==`는 `False` → 이건 **충돌**일 뿐 규약 위반은 아님. 허용됨. 단지 성능 손해(같은 버킷에 더 많은 원소가 몰려 조회가 느려짐)

**결론**: `__eq__`에 쓴 필드의 **딱 그 집합**을 `__hash__`에서도 써야 합니다. 덜 쓰면 성능 손해(허용), 더 쓰면 규약 위반(금지).

#### "그럼 클래스의 메서드나 다른 attribute는?"

메서드와 attribute의 본성 차이를 보면 답이 나옵니다.

- **메서드**는 값이 아니라 **코드**입니다. `entry.from_line`은 어떤 인스턴스에서 접근하든 같은 함수 객체를 가리킵니다(바운드 메서드 래퍼는 매번 새로 만들어지지만 내부의 함수는 동일). 메서드는 "인스턴스를 구분하는 정보"가 아니라 "모든 인스턴스가 공유하는 동작"이에요. **동등성 판단에 쓸 게 없습니다**
- **attribute 중 어떤 것이 "식별에 참여하는가"는 설계 결정**입니다. 예컨대 어떤 VO에 `_cache: dict` 같은 **내부 캐시**가 있다면, 그건 동등성에 포함되면 안 됩니다 — 같은 값을 가진 두 인스턴스가 캐시 상태만 다르다고 "다른 것"이 되면 VO의 본성이 깨집니다
- 그래서 파이썬은 **"어느 필드가 식별에 참여하는지"를 프로그래머가 명시적으로 고르게** 합니다. 자동으로 "모든 attribute 다 써"가 아니라

**사고 실험**: ListEntry에 `_debug_created_at: float` 같은 임시 디버그 필드가 추가됐다고 상상해 보세요. 이걸 `__hash__`에 넣으면 같은 audio_path/speaker/language/text를 가진 두 인스턴스가 다른 해시를 갖게 되어 dict 키로 쓸 수 없게 됩니다. "디버그 타임스탬프가 다르다"는 이유로 값은 같은 두 객체를 다른 것으로 취급하는 건 원하는 동작이 아닙니다.

#### `hash(tuple(...))`의 메커니즘

```python
return hash((self.audio_path, self.speaker, self.language, self.text))
```

이 한 줄이 하는 일:

1. 네 필드로 **튜플 객체**를 만든다
2. 파이썬의 tuple `__hash__`가 내부적으로 **각 원소의 해시를 재귀적으로 계산**하고 섞어서 최종 정수를 뽑는다 (XOR + 곱셈 기반, 구현은 CPython 내부)
3. 이 정수가 결과

즉 "4개 값을 한 덩어리로 지문화"입니다. 튜플을 쓰는 이유는 파이썬이 이미 "여러 값의 조합을 해싱"하는 방법을 튜플 해싱에 심어 놨기 때문이에요. 직접 비트 섞기를 할 필요가 없습니다. **관용구**:

```python
def __hash__(self):
    return hash((self.f1, self.f2, self.f3, ...))
```

**주의**: 튜플 안의 원소는 전부 **hashable**해야 합니다. 문자열·정수·불리언·frozenset·다른 튜플은 OK. **리스트·dict·set은 unhashable** — 이 중 하나가 필드로 들어오면 `__hash__` 내부에서 `TypeError: unhashable type: 'list'`가 납니다. 지금 `ListEntry`는 4필드 모두 `str`이라 문제 없음. Phase 03 이후 `SliceOptions` 같은 VO가 등장할 때 "옵션 리스트"를 담으면 이 지점이 아프게 돌아옵니다 — `tuple(옵션_리스트)`로 변환해서 해시하거나, 필드 자체를 처음부터 튜플로 저장하는 식으로 해결.

#### 해시에 대한 마지막 포인트 — "내부 숫자를 신경 쓰지 말 것"

해시 값 자체의 정수는 **아무런 의미가 없습니다**. `hash("hello")`가 실행할 때마다 바뀔 수도 있어요(파이썬은 보안 목적으로 실행마다 문자열 해시를 랜덤화합니다). 중요한 건 **한 프로세스 안에서 일관성이 유지된다는 것**과 **`==`와 정합한다는 것**, 이 두 가지뿐.

그래서 해시를 볼 때 "어떤 숫자가 나왔지?"를 궁금해 하지 말고, **"`==`가 같다고 할 것들을 전부 같은 숫자로 보내고 있는가?"**만 생각하면 충분합니다. 그게 전부입니다.

### Q: `__eq__`는 딱히 hash 등의 내용을 비교하는 것 같지 않고 평이한 필드 비교인데, 저희 테스트가 실패한 지점의 `a == b`는 이 부분이 아닌가요? 그런데 테스트 실패 메시지에는 메모리 주소값이 나왔고, 여기는 언뜻 봤을 때 평이한 비교를 하는 — 이 괴리는 어디서 오는 건가요?

정확히 짚은 괴리입니다. 세 가지 **별개의 메커니즘**이 **우연히 전부 "주소"에 관련**되어 있어서 하나의 덩어리처럼 보이는 현상이에요. 풀어 보면:

#### 메커니즘 ① — 기본 `__eq__` 자체가 주소 비교

수동 `__eq__`를 쓰기 **전**의 `ListEntry`는 `object`의 기본 `__eq__`를 상속했습니다. 그 구현(의사 코드):

```python
# object.__eq__ (의사 코드)
def __eq__(self, other):
    return self is other   # ← 메모리 주소가 같은가?
```

`is` 연산자는 **정확히 메모리 주소(엄밀히는 `id()` 값)가 같은지**를 묻습니다. `a`와 `b`는 **따로 만들어진 두 인스턴스**라 `id(a) != id(b)` → `a is b`는 `False` → 기본 `==`도 `False`. 이 단계에서 이미 테스트는 실패합니다.

**여기가 첫 번째 "주소"**: 비교 로직 자체가 주소 기반.

#### 메커니즘 ② — 기본 `__repr__`이 주소를 문자열에 포함

모든 객체의 기본 문자열 표현(`repr(obj)`)은 이런 형태입니다:

```
<voxprep.parsing.list_file.ListEntry object at 0x108facd10>
```

여기 `0x108facd10`이 `id(obj)`의 16진수 — **메모리 주소**. 이건 파이썬의 모든 객체 기본 표시 형식입니다. 사용자가 따로 `__repr__`을 오버라이드하지 않으면 전부 이 모양이에요.

**여기가 두 번째 "주소"**: 객체를 화면에 찍는 방식이 주소 기반.

#### 메커니즘 ③ — pytest의 assertion 실패 포맷이 `repr()`을 사용

pytest는 `assert a == b`가 실패하면 대략 이 형식으로 출력합니다:

```
>       assert a == b
E       assert <repr(a)> == <repr(b)>
E         +  where a = <repr(a)>
E         +  and b = <repr(b)>
```

값을 보여줄 때 **`repr()`을 호출**합니다. 그리고 `ListEntry`엔 `__repr__`이 없으니 기본 `<... at 0x...>`로 찍혀요.

**여기가 세 번째 "주소"**: 화면 출력 포맷이 `repr`을 쓰고, `repr`이 주소를 포함.

#### 세 메커니즘의 완전히 독립된 관계

- ①·②·③은 **각자 따로 존재하는 별개의 파이썬 기능**입니다
- 기본값에서는 셋 다 "주소"에 묶여 있어서 한 덩어리처럼 보이지만, 각각을 따로 오버라이드할 수 있어요
- **사용자가 수동 `__eq__`를 정의한 후로는 ①이 사라집니다** — 비교가 주소 기반에서 필드 기반으로 바뀜. ②와 ③은 그대로. 그래서 이제는 **비교는 필드로, 화면 출력은 주소로** 찍힙니다. 이게 pytest 에러 메시지의 모순처럼 보이는 것의 원인

#### 증명하는 실험

GREEN 상태에서 REPL을 띄워 확인:

```bash
uv run python
```

```python
>>> from voxprep.parsing.list_file import ListEntry
>>> a = ListEntry("a.wav", "s", "ko", "hi")
>>> b = ListEntry("a.wav", "s", "ko", "hi")
>>> a == b                  # True  ← 수동 __eq__가 필드를 봄
>>> a is b                  # False ← 주소는 다름
>>> repr(a)                 # '<voxprep...ListEntry object at 0x...>' ← 기본 __repr__
>>> id(a), id(b)            # 서로 다른 숫자 ← 분명히 다른 객체
>>> hash(a) == hash(b)      # True  ← 수동 __hash__가 필드 튜플을 씀
>>> {a, b}                  # {<...>}  ← set에 두 개 넣었는데 한 개만 남음 (VO 동작 확인)
```

`a == b`는 `True`인데 `a is b`는 `False`. 이걸 직접 찍어 보면 **"같다"와 "같은 객체다"가 별개 개념**이라는 게 손에 잡힙니다. Java의 `equals` vs `==`와 정확히 같은 구도예요.

#### pytest 에러 메시지를 올바르게 읽는 법

이후로 `assert a == b` 실패 메시지에 주소가 나와도 당황하지 마세요. 주소는 **"두 객체가 다른 인스턴스라는 힌트"**일 뿐, 비교 로직이 주소를 본다는 뜻이 아닙니다. 실제 비교가 뭘 봤는지는 **`__eq__` 구현을 읽어야** 알 수 있어요.

**더 나은 진단을 원하면**: `__repr__`도 구현해서 `ListEntry(audio_path='a.wav', speaker='s', ...)` 같은 의미 있는 문자열이 나오게 하면, 에러 메시지가 이렇게 바뀝니다:

```
assert ListEntry(audio_path='a.wav', ...) == ListEntry(audio_path='a.wav', ...)
```

그럼 "두 객체가 값은 같아 보이는데 왜 다르다고 하지?"의 혼동이 없어져요. 이건 dataclass로 이관하면 자동으로 해결됩니다(그때 `__repr__`도 공짜로 딸려옴).

### 두 질문의 공통 교훈

두 질문 모두 **"파이썬 객체의 기본 동작은 주소 기반"**이라는 사실을 중심에 놓고 이해하면 한 번에 풀립니다. 그리고 **각 메서드(`__eq__`, `__hash__`, `__repr__`)는 독립적으로 오버라이드 가능**한데, 기본값이 전부 주소 기반이라 오버라이드 전엔 하나처럼 보이고, 오버라이드 후엔 조합이 어긋나 보입니다.

요점 셋:

1. **해시는 `==`와 동일한 필드 집합을 사용해야 한다** — 파이썬 해시 규약(`a == b → hash(a) == hash(b)`)의 직접적 귀결
2. **비교(`==`)와 표현(`repr`)은 별개의 메서드** — 하나 바꿨다고 다른 게 따라 바뀌지 않음. 기본값이 전부 주소 기반이라 그렇게 보였을 뿐
3. **dataclass는 이 셋을 한 번에 일관되게 생성해 준다** — (B) 수동 구현 후 (A) dataclass로 이관할 때 가장 크게 느껴질 이득. 특히 `__repr__`이 자동으로 의미 있게 바뀌어 디버깅 메시지가 돌변함
